## Payment statuses

Every payment moves through the following lifecycle. Status codes are stored in the `status` field of the `Payment` record.

| Status code | Spanish label | Description |
|---|---|---|
| `pending_validation` | Pendiente validación | Payment registered but not yet reviewed by the tesorero |
| `validated` | Validado | Tesorero confirmed the payment is correct |
| `confirmed` | Confirmado | System-confirmed state (treated as validated in the UI) |
| `rejected` | Rechazado | Tesorero rejected the payment |
| `reversed` | Revertido | A previously validated payment was reversed; not deleted from history |

<Note>
  Reversed payments remain in the history with a `reversed` status and an optional reversal reason. All fee allocations and credit generated by the payment are undone when it is reversed.
</Note>

## Payment data fields

The following fields are available on a `Payment` record.

| Field | Type | Description |
|---|---|---|
| `id` | `string` | Unique payment identifier |
| `player_id` | `string` | ID of the player who made the payment |
| `player_name` | `string \| null` | Display name of the player |
| `amount` | `number` | Base payment amount in CLP |
| `amount_total` | `number \| null` | Total amount (may differ from `amount` if adjusted) |
| `currency` | `string` | Always `CLP` |
| `status` | `string` | One of the status codes above |
| `transfer_ref` | `string \| null` | Transfer reference number entered by the player |
| `reference_number` | `string \| null` | Alternative reference field |
| `notes_player` | `string \| null` | Notes submitted by the player at registration |
| `notes` | `string \| null` | General notes |
| `notes_treasurer` | `string \| null` | Notes added by the tesorero during validation or rejection |
| `payment_method` | `string \| null` | Payment method (e.g. `transfer`) |
| `payment_date` | `string \| null` | Date the payment was made (ISO date) |
| `created_at` | `string` | Timestamp when the record was created |
| `target_month` | `string \| null` | Month the payment applies to, in `YYYY-MM` format |
| `tournament_id` | `string \| null` | Tournament the payment belongs to |
| `tournament_name` | `string \| null` | Display name of the tournament |
| `series_id` | `string \| null` | Series associated with the payment |
| `series_name` | `string \| null` | Display name of the series |
| `receipt_file_id` | `string \| null` | ID of the uploaded receipt file, if any |
| `reversed_at` | `string \| null` | Timestamp of reversal |
| `reversed_by_user_id` | `string \| null` | ID of the user who reversed the payment |
| `reversal_reason` | `string \| null` | Reason provided at reversal |
| `player_debt_blocked` | `boolean` | Whether the player has an active debt block in the payment's tournament |

## Self-service payment registration

Any authenticated user can register a payment on their own behalf from the **Dashboard** page. The request goes to `POST /api/payments/self-register` and uses the player's RUT to look up their account.

```json
POST /api/payments/self-register
{
  "tournament_id": "<tournament_id>",
  "rut": "<normalized_rut>",
  "amount": 15000,
  "target_month": "2026-03",
  "reference_number": "123456789",
  "notes_player": "123456789"
}
```

Required fields: `tournament_id`, `rut`, `amount`. `target_month` is optional but recommended; if omitted the payment is not linked to a specific month.

The endpoint returns `{ "id": "<payment_id>" }`. Use the returned `id` to upload a receipt.

<Tip>
  Before submitting, the dashboard calls `GET /api/fees/player-fee-by-rut?rut=<rut>&year_month=<YYYY-MM>&tournament_id=<id>` to pre-fill the suggested fee amount. The response includes `fee_amount` and `fee_source` (`player`, `series`, or `general`).
</Tip>

## Receipt upload

After registering a payment, you can attach a receipt file:

```
POST /api/payments/:id/receipt
Content-Type: multipart/form-data
```

Send the file as a multipart upload. The `receipt_file_id` field on the payment record is populated once the upload succeeds.

Tesorero users can view the receipt from the **Pendientes** tab by clicking **Ver comprobante** next to a payment.

## Tesorero payment registration

Users with the `tesorero` or `admin` role can also register payments directly from the treasury module using the **Registrar pago** button. This uses `POST /api/payments` with a `player_id` (instead of a RUT) and the full payment details.

```json
POST /api/payments
{
  "player_id": "<player_id>",
  "amount": 15000,
  "currency": "CLP",
  "transfer_ref": "123456789",
  "notes_player": "123456789",
  "target_month": "2026-03",
  "tournament_id": "<tournament_id>"
}
```

## Validation workflow

After a player registers a payment, the tesorero reviews it in the **Pendientes** tab and takes one of three actions.

<Steps>
  <Step title="Player registers payment">
    The player submits a payment via the Dashboard (self-register) or the tesorero registers it directly. The payment is created with status `pending_validation`.
  </Step>
  <Step title="Tesorero reviews">
    Open the **Pendientes** tab in the treasury module. Each card shows the player name, tournament, amount, target period, reference number, and a link to the receipt if one was uploaded.
  </Step>
  <Step title="Validate or reject">
    Click **Validar** to approve the payment. A modal prompts for an optional **Nota del tesorero** (`notes_treasurer`). If the player has an active debt block in the payment's tournament, you can optionally release the block at the same time.

    Click **Rechazar** to reject the payment. Provide an optional note explaining the reason. The status changes to `rejected`.
  </Step>
  <Step title="Payment counted in collection">
    A validated payment (status `validated` or `confirmed`) is included in the **Recaudado** total and applied to the player's fee charges for the target month.
  </Step>
</Steps>

## Payment reversal

A `tesorero` or `admin` user can reverse any payment with status `validated` or `confirmed` from the **Historial** tab.

```
POST /api/payments/:id/reverse
{ "reason": "Pago duplicado" }
```

The `reason` field is optional but recommended for audit purposes. After reversal:

- The payment status changes to `reversed`.
- All fee charge allocations linked to this payment are undone.
- Any credit balance generated by the payment is removed.
- The payment remains visible in the history with its `reversed_at` timestamp and `reversal_reason`.

<Warning>
  Reversal cannot be undone. If a payment was reversed by mistake, register a new payment for the same amount and period.
</Warning>

## Payment history

The **Historial** tab shows all payments regardless of status. Use the following filters to narrow the list:

- **Torneo** and **Serie** selectors in the page header
- Quick-period buttons: **Todo**, **Este mes**, **Mes pasado**, **Últimos 3 meses**
- Custom date range using the **desde / hasta** date inputs

The history table shows: player name, series, tournament, payment date (or creation date if no payment date), target period, amount, and status badge.

## Roles

| Action | Roles |
|---|---|
| Self-register a payment | All authenticated roles (`jugador`, `delegado`, `tesorero`, `admin`) |
| Upload a receipt | All authenticated roles |
| View all payments (Historial) | `tesorero`, `admin` |
| Validate a payment | `tesorero`, `admin` |
| Reject a payment | `tesorero`, `admin` |
| Reverse a payment | `tesorero`, `admin` |
| Register a payment via treasury (by player ID) | `tesorero`, `admin` |

## API endpoints

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/payments` | List payments; supports `status`, `series_id`, `tournament_id`, `date_from`, `date_to`, `limit` query params |
| `POST` | `/api/payments` | Register a payment (tesorero/admin, uses `player_id`) |
| `POST` | `/api/payments/self-register` | Register a payment (any role, uses `rut`) |
| `POST` | `/api/payments/:id/receipt` | Upload a receipt file (multipart) |
| `POST` | `/api/payments/:id/validate` | Validate a pending payment |
| `POST` | `/api/payments/:id/reject` | Reject a pending payment |
| `POST` | `/api/payments/:id/reverse` | Reverse a validated payment |
